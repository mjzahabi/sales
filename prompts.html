A complete guide to prompt types and patterns — what they are, how they work, when to use them, and real examples for each.

📐

Prompt Types

The three structural positions in an LLM conversation where you inject text

Core principle: An LLM has no memory, no world model, and no intent of its own. It only has the tokens in front of it. Every prompt type is a different position in that token stream — a different moment where you can shape what the model sees before it generates a response.
System

System Prompt

The persistent background context set before any conversation. Defines role, behavior rules, tone, and scope. Every user message is processed through this lens.

Analogy: The employee handbook handed to a new hire before day one — always in the background, always in effect.
User

User Prompt

The actual message in the conversation — your question, task, or instruction. The live request that the model responds to, within the context the system prompt established.

Analogy: Each individual task slip dropped on the employee's desk during the day.
Assistant

Assistant Prefill

Pre-populating the start of the model's response before generation begins. Forces the output to begin in a specific way — locking in format, structure, or tone from word one.

Analogy: Handing someone a sentence already started and asking them to finish it their way.
🧱

System Prompt

The persistent behavioral foundation — set once, applies to every turn

System prompts are the most powerful lever you have. They establish identity, rules, tone, scope, and output format before a single user message arrives. Think of it less like a command and more like a role brief — the model will interpret everything that follows through this lens.

AnalogyImagine onboarding a brilliant contractor. Before they start, you hand them a doc: "You work exclusively on automotive topics. Always respond professionally. Never discuss competitors. When unsure, ask for clarification." Every question they get, they'll filter through that framing.
Example — Internal Support AgentSystem
// SYSTEM PROMPTYou are a support agent for VinSolutions CRM. Help dealership staff troubleshoot issues and optimize workflows. RULES: - Only answer questions about VinSolutions and dealership CRM processes - Always ask for the dealer's Account ID before troubleshooting - Use clear, jargon-free language - If you don't know something, say so — never guess OUTPUT FORMAT: 1. Direct answer (1–2 sentences) 2. Step-by-step instructions if applicable 3. Link to documentation if available
Example — AI Sales Trainer PersonaSystem
// SYSTEM PROMPTYou are Alex, a senior automotive sales trainer with 20 years coaching BDC teams. Your style is direct, encouraging, and grounded in real dealership scenarios. CONSTRAINTS: - Keep responses under 200 words unless the user asks for more - Always frame advice around real scenarios, not theory - Never use buzzwords like "synergy" or "leverage" as verbs - If asked about competitors, redirect to best practices instead
Use CaseWhat to Define in System Prompt
Customer-facing chatbotPersona, tone, scope limits, escalation rules, what NOT to say
Internal AI assistantTeam context, available data sources, preferred output formats
Document generatorOutput format spec (JSON, Markdown, HTML), field names, length rules
Code assistantLanguage, style guide, commenting conventions, testing requirements
Objection handler agentProduct context, known objection categories, tone, escalation path

User Prompt

The live message — your task, question, or instruction in the moment

The user prompt is the conversational turn. While the system prompt is the standing brief, the user prompt is the specific task. Strong user prompts are explicit about three things: what to do, what context to use, and what to produce.

Vague — low quality output❌ Weak
Write me an email about the deal.
Specific — high quality output✓ Strong
Write a 3-sentence follow-up email to a dealership GM who attended our Xtime demo yesterday but hasn't responded. Tone: professional but warm. Goal: schedule a 30-min follow-up call this week. Do not mention pricing.
Anatomy of a strong user promptUser
[TASK] Summarize the following sales call transcript. [CONTEXT] This is a call between our Product Consultant and a Chevy dealer's GSM discussing VinSolutions renewal. The dealer raised concerns about ROI and response time from support. [FORMAT] Output as: - 1 sentence TL;DR - 3 bullet objections raised - 1 recommended next step [CONSTRAINTS] Under 150 words. Plain language. No jargon. [TRANSCRIPT] ...
Pro tip: Use positive instructions over negative ones when possible. "Respond in plain language" gets better results than "Don't use jargon." The model steers toward what it's told to do more reliably than away from what it's told to avoid.

Assistant Prefill

Seed the start of the model's response to lock in format or structure

Prefill injects text at the beginning of the assistant's turn before generation starts. The model continues from that point — making it powerful for forcing JSON output, eliminating preamble like "Sure, I'd be happy to help!", or locking in a specific document structure. Available in Anthropic Claude and Mistral APIs.

AnalogyLike handing someone a form with the first few fields already filled in. They don't reconsider the format — they just pick up where you left off and fill in the rest.
Without prefill
User: Analyze this objection and respond. Assistant: Sure! I'd be happy to help you with that objection. Let me take a look at what you've shared and provide some thoughts... // ← 30 words of preamble before useful content
With prefill
User: Analyze this objection and respond. Assistant (prefilled): { // ← Model continues directly in JSON structure
Example — Force JSON output with prefillAssistant
// userAnalyze this sales objection and return a structured response. // assistant (prefilled — model continues from here){ "objection_type": "...model fills in from here
Availability note: Prefill is an API-level feature. In chat interfaces like Claude.ai, you can simulate it by ending your user message with the text you want the response to start with — asking the model to "start your response with {" and then continue in valid JSON works similarly.

Zero-Shot Prompting

Just ask — no examples, no scaffolding, no demonstrations

Zero-shot is the default mode. You state the task and the model draws entirely on its training to respond. It works well for capable models on clear, common tasks. When it fails, the fix is usually to add examples (few-shot) or reasoning steps (chain-of-thought) rather than to rephrase the same zero-shot prompt over and over.

AnalogyAsking a new hire to complete a task by simply describing what you want — trusting that their experience and judgment are enough to get it right without seeing an example first.
Zero-Shot ExampleZero-Shot
// userClassify the sentiment of this customer review as Positive, Neutral, or Negative. Review: "The setup took forever and support never called back, but once it was running the reporting was actually pretty useful." // model responseMixed (leans Negative) — The review acknowledges a positive outcome (useful reporting) but leads with two clear pain points (slow setup, unresponsive support). Sentiment is mixed with a negative lean.
Zero-Shot — Automotive ExampleZero-Shot
// userSummarize the following dealer objection in one sentence, then suggest the best product feature to address it. Objection: "We already have a process for this — my team's been doing it the same way for 10 years and they're not going to change." // model responseThe dealer is resistant to workflow change driven by long-standing team habits. Best feature to highlight: workflow integration and minimal-disruption onboarding — emphasize that the tool works alongside existing processes, not against them.
Best for: Simple classification Summarization Translation General Q&A Well-defined tasks

Few-Shot Prompting

Show examples of the exact pattern you want, then make the real request

Few-shot prompting provides 2–5 input/output pairs that demonstrate the pattern before your actual request. The model learns the format, style, and logic from the examples rather than from explicit instruction. It's especially powerful for custom classification schemes, structured outputs, or any task where "doing it right" is hard to describe but easy to show.

AnalogyInstead of writing a 3-page style guide, you hand a new employee three completed examples of the report and say "produce one that looks like these." The examples carry more information than any description could.
Few-Shot — Custom Objection ClassifierFew-Shot
// user — 3 examples, then the real request EXAMPLE 1 Objection: "We're locked into a contract with another vendor until Q4." Category: TIMING EXAMPLE 2 Objection: "My sales team won't actually use another tool — adoption always falls off." Category: ADOPTION EXAMPLE 3 Objection: "We tried CRM before and it didn't move the needle on sales." Category: ROI_SKEPTICISM NOW CLASSIFY: Objection: "The price is fine but I need to get my GM and Controller to sign off first." Category: STAKEHOLDER_APPROVAL
Quality tip: Few-shot examples should be diverse enough to cover edge cases. If all your examples look the same, the model over-fits to that pattern. Aim for variety in phrasing and scenario while keeping the output format consistent.
Best for: Custom classification Proprietary output formats Style matching Data extraction schemas Tone calibration

Chain-of-Thought (CoT)

Ask the model to reason step-by-step before reaching a conclusion

Chain-of-thought prompting instructs the model to walk through its reasoning before producing the final answer. This dramatically improves accuracy on complex problems — math, logic, multi-step analysis, and anything where intermediate steps matter. The model "shows its work" and catches errors in reasoning that would otherwise slip through to a wrong conclusion.

AnalogyThe difference between asking a student "what's the answer?" versus "show your work, then give the answer." The latter catches errors at every step instead of just validating the final number.
Without CoT — reasoning hidden
User: Our MSS team closed 47 deals this quarter. Each deal averages $1,200 MRR. If churn is 8% annually, what's expected net new MRR after 12 months? Assistant: Your expected net new MRR after 12 months is approximately $52,704. // ← No reasoning shown. Is this right? Hard to verify.
With CoT — reasoning visible
User: ...same question. Think step by step. Assistant: Step 1: Gross MRR from new deals 47 × $1,200 = $56,400 Step 2: Annual churn rate → monthly 8% / 12 = 0.667% per month Step 3: MRR after 12 months of churn $56,400 × (1 - 0.00667)^12 ≈ $51,631 Step 4: Net new MRR ≈ $51,631
CoT Trigger PhrasesCoT
// Drop any of these into your prompt to activate chain-of-thought "Think step by step." "Walk me through your reasoning before giving the answer." "Let's think through this carefully." "Before answering, reason through each step." "Show your work."
Best for: Multi-step math Logic puzzles Complex analysis Debugging decisions Risk assessment

Role Prompting

Assign a persona or expert identity to prime the model's framing and vocabulary

Role prompting assigns the model an identity or expert persona before it responds. This shapes tone, vocabulary, knowledge emphasis, and even risk tolerance. A "senior engineer" and a "cautious legal reviewer" will approach the same document very differently — and both responses are more useful than a generic one.

AnalogyBefore asking a question, you introduce yourself to the right expert in the room. "I'm going to ask Sarah, our CFO" versus "I'm going to ask Marcus, our sales trainer." Same question — completely different useful answer.
Role Prompting ExamplesRole
// Shifts framing + vocabulary significantly ADVERSARIAL REVIEWER: "Act as a skeptical dealership GM who has been burned by software vendors before. Review this product pitch and tell me every objection I should expect." DOMAIN EXPERT: "You are a fixed ops director with 15 years running service departments. What are the three biggest workflow bottlenecks in a high-volume service lane?" EDITOR / CRITIC: "You are a ruthless editor who cuts everything non-essential. Review this executive summary and cut it to under 100 words without losing any key insight." PERSONA FOR EMPATHY: "You are a first-year service advisor at a family-owned dealership. Describe how you would feel encountering this new scheduling software on day one."
Best for: Expert-level analysis Adversarial review Audience empathy Tone calibration Red-teaming content

Structured Prompting (ICF)

Explicit Instruction + Context + Format — the well-written project brief

Structured prompting separates your request into distinct labeled components: what to do, what background knowledge to use, and how to format the output. It's the difference between a vague Slack message and a well-written project brief. The model doesn't have to infer any of these — you've made all three explicit.

AnalogyA contractor gets two types of requests. The first: "build something nice in the backyard." The second: "build a 12x16 cedar deck with three steps, railing on two sides, stained in dark walnut, and completed before June 15." The second request produces exactly what you wanted.
Structured Prompt — Coaching EmailICF
INSTRUCTION: Write a coaching note from me to a Product Consultant on my team who missed their demo-to-close conversion goal this quarter. CONTEXT: - Target was 35% demo-to-close; actual was 24% - The rep is high energy and great at rapport, but struggles with discovery questioning - This is our first formal conversation about the gap - I want to be direct but encouraging — not punitive FORMAT: - Opening: acknowledge strengths (1 sentence) - Core observation: the gap and what's driving it (2–3 sentences) - Action item: one specific coaching commitment - Closing: forward-looking, not punitive - Max 150 words total
Best for: Document generation Agent prompts Constrained outputs Repeatable workflows Production prompts

Retrieval-Augmented Generation (RAG)

Dynamically inject relevant source material into the prompt before the model responds

RAG is a system design pattern, not something you manually write. A retrieval layer fetches relevant documents, knowledge base entries, or data — and injects them into the prompt automatically before the model responds. The model becomes a researcher who's handed a pre-filtered folder of relevant files right before answering your question.

AnalogyInstead of trusting a consultant to remember everything from memory, you give them a searchable library and a rule: "before answering any question, pull the 3 most relevant documents and use those as your primary source." Their answer is grounded in current, specific information — not just general knowledge.
What a RAG prompt looks like at runtimeRAG
// system — instructs model how to use injected docs You are a product support assistant. Answer questions using ONLY the documentation provided below. If the answer isn't in the documents, say "I don't have that information" rather than guessing. --- RETRIEVED DOCUMENTS --- [DOC 1: VinSolutions Lead Response Settings — Last updated 2024-11] "Lead response time windows can be configured under Settings > BDC > Lead Routing..." [DOC 2: Xtime Scheduler Integration FAQ — Last updated 2025-01] "When VinSolutions and Xtime are integrated, appointment creation syncs within 90 seconds..." -------------------------- // user How long does it take for an Xtime appointment to show up in VinSolutions?
RAG is right when...RAG is overkill when...
Answers need to be grounded in current docsThe model's training data is sufficient
Knowledge changes frequently (release notes, pricing)Questions are general and stable
Accuracy and citation matter (support, compliance)Creative tasks where grounding isn't required
You have a large knowledge base (> model context window)Small fixed context can just be pasted in directly

Self-Consistency

Generate multiple independent reasoning paths, then pick the most common answer

Self-consistency runs the same prompt multiple times with slightly varied reasoning, then aggregates the outputs — typically picking the most common answer. It treats the model like a panel of independent reviewers: if 4 out of 5 reach the same conclusion through different paths, you have much higher confidence in that answer.

AnalogyYou need to estimate a project timeline. Instead of asking one person, you ask five different senior engineers independently. You don't average the numbers — you notice that four of them land in the 6–8 week range and one outlier says 16 weeks. The consensus wins.
Self-Consistency — High-Stakes AnalysisSelf-Consistency
// Prompt run 3x with high temperature, then outputs compared PROMPT (same for all 3 runs): A dealer has used VinSolutions for 18 months. Their CRM lead response time averages 4.2 hours. Industry benchmark is under 1 hour. They're asking if they should renew. Reason through whether renewal makes sense from the dealer's perspective. Think step by step. RUN 1 CONCLUSION: Renewal is risky — the tool isn't driving behavioral change. RUN 2 CONCLUSION: Renewal with conditions — needs a structured coaching plan first. RUN 3 CONCLUSION: Renewal with conditions — the gap is process, not product. Majority consensus: Renewal with conditions (2/3) → proceed with that framing
Best for: High-stakes decisions Ambiguous analysis Complex reasoning When errors are costly

Tree of Thought (ToT)

Explore multiple reasoning branches simultaneously, evaluate each, then converge

Tree of Thought extends chain-of-thought into a branching structure. Instead of one linear reasoning path, the model generates multiple candidate approaches, evaluates each, prunes weak branches, and converges on the most promising one. It's CoT with a deliberate search process layered on top.

AnalogyA chess player doesn't just calculate one sequence of moves — they mentally branch out multiple lines, evaluate each a few moves deep, discard the weaker ones, and commit to the line that holds up best under scrutiny. ToT makes this explicit in the prompt.
Tree of Thought — Strategy DecisionToT
// user We need to increase demo-to-close conversion for the MSS team. Generate 3 distinct strategic approaches. For each: explain the approach, identify the key risk, and score it 1–10 on feasibility and impact. Then recommend the best one and explain why. // model output BRANCH A — Discovery-first coaching: Approach: Retrain PCs on discovery questioning before demos Risk: Takes 4–6 weeks before results show Feasibility: 8/10 | Impact: 9/10 BRANCH B — Demo personalization toolkit: Approach: Build a pre-demo research template tied to dealer profile Risk: Adoption depends on PC discipline Feasibility: 7/10 | Impact: 7/10 BRANCH C — Compressed deal cycles via urgency framing: Risk: Can feel pushy, damages trust Feasibility: 6/10 | Impact: 5/10 Recommendation: Branch A — highest impact with manageable timeline. Combine with Branch B as a supporting tool.
Best for: Strategic planning Architecture decisions Complex problem solving Option comparison

ReAct Prompting

Interleave reasoning and action in a loop — the engine behind modern AI agents

ReAct (Reasoning + Acting) is the pattern where the model alternates between Thought (reasoning about what it knows and needs), Action (calling a tool, searching, running code), and Observation (reading the result) — looping until it can produce a grounded answer. It's the backbone of virtually every AI agent in production today, including M365 Copilot agents.

AnalogyThe difference between a consultant who answers from memory versus one who stops mid-conversation, checks a live data source, reads the result, reasons again, and then answers. The second consultant is doing ReAct — and their answer is grounded in reality, not just training.
ReAct Loop — Live ExampleReAct
// userWhat is the current industry benchmark for lead response time in automotive CRM? Thought: I don't have current 2025 data in memory. I should search for it. Action: search("automotive CRM lead response time benchmark 2025") Observation: Top performers respond under 1 hour; industry average is 3.5 hours. Thought: I have current data now. I can answer with a citation. Answer: The current benchmark for top-performing dealers is under 1 hour...
ReAct System Prompt Template — For Agent BuilderSystem
You are an objection handling assistant for the MSS team. When a user submits a dealer objection: 1. THINK: Identify the objection category (pricing, timing, adoption, ROI, competition) 2. ACT: Search the knowledge base for matching objection responses 3. OBSERVE: Review what was returned — is it relevant and current? 4. RESPOND: Synthesize a 3-part response: validate, reframe, evidence If the knowledge base returns nothing useful, say so and escalate.
Best for: AI agentsTool-using modelsLive data retrievalMulti-step workflowsGrounded answers

Question Refinement Pattern

Ask the model to suggest a better version of your question before answering it

Instead of answering your question as-is, the model first proposes a sharper, more precise version of it — then either asks you to confirm or answers that improved version automatically. It's a built-in quality check that surfaces assumptions you didn't know you were making.

AnalogyYou ask a doctor "what's wrong with me?" A good doctor doesn't just answer that — they ask clarifying questions and reframe: "I think what you're really asking is whether these symptoms indicate X. Let me address that." The reframe produces a far more useful answer.
Question Refinement — FormatPattern
// TemplateWhenever I ask a question, suggest a better version of my question and ask me if I'd like you to answer that version instead. // User asksHow do I improve my team's performance? // Model respondsA more precise version of your question might be: "What are 3 high-leverage coaching interventions for a 50-person product consultant team that is underperforming on demo-to-close conversion by more than 10 points?" Would you like me to answer this version instead?
Best for: Exploratory researchAmbiguous tasksStrategy conversationsCoaching tools

Cognitive Verifier Pattern

Break a complex question into sub-questions, answer each, then synthesize

The model generates a set of smaller questions needed to fully answer your original question, answers each one individually, and then combines the answers into a final response. It's a structured way to handle questions that carry hidden complexity — where jumping straight to an answer means skipping critical sub-problems.

AnalogyA good consultant doesn't answer "should we expand to this market?" in one breath. They decompose it: What's the TAM? Who's the competition? What's our distribution fit? What's the regulatory landscape? They answer each, then synthesize a recommendation. That's cognitive verification.
Cognitive Verifier — FormatPattern
// Trigger phrase in system or user promptWhen answering a question, generate a list of sub-questions that need to be answered first. Answer each sub-question, then use those answers to construct your final response. // User asksShould we build our objection handler agent in Copilot Agent Builder or as a custom Salesforce Agentforce agent? // Model breaks it down Sub-Q 1: Where does the team currently work — M365 or Salesforce? Sub-Q 2: What data sources does the agent need to access? Sub-Q 3: What are the IT governance constraints for each platform? Sub-Q 4: What is the expected maintenance burden for each option? → Synthesized recommendation based on all four answers...
Best for: Complex decisionsStrategy analysisResearch tasksBuild vs. buy

Audience Persona Pattern

Tell the model who it's talking to so it calibrates language and depth appropriately

Where Role Prompting tells the model who to be, Audience Persona tells the model who it's talking to. The same explanation of a technical concept should land very differently for a first-year BDC rep versus a Chief Technology Officer. This pattern ensures the model calibrates vocabulary, assumed knowledge, and depth to the actual reader.

AnalogyA great presenter shifts their material depending on who's in the room. The same product update gets pitched differently to a dealer's GM versus their IT director — same facts, completely different framing, vocabulary, and emphasis.
Audience Persona — ExamplesPattern
// In system or user promptExplain the following as if you are talking to [audience]. EXAMPLE 1 — Non-technical executive: "Explain how RAG works to a dealership GM who has never heard of AI and is skeptical of technology vendors. No acronyms, no tech jargon." EXAMPLE 2 — Technical audience: "Explain the tradeoffs between vector similarity search and BM25 retrieval to a senior ML engineer evaluating RAG architectures." EXAMPLE 3 — Sales coaching: "Explain the value of CRM lead routing to a first-year BDC rep who thinks their manual process is working fine."
Best for: Training materialsExec communicationsCustomer-facing contentMulti-audience docs

Flipped Interaction Pattern

Let the model ask you the questions instead of you asking the model

Instead of you driving the conversation, you tell the model what you're trying to achieve and instruct it to interview you — asking the questions it needs answered in order to help you well. This is especially powerful when you don't know what information is relevant, or when you're exploring a problem space you haven't fully mapped yet.

AnalogyInstead of filling out a form yourself, you sit down with an expert who asks you the right questions, knows what follow-ups matter, and builds a complete picture through dialogue. You provide the answers; they provide the structure.
Flipped Interaction — FormatPattern
// TemplateI want you to help me [goal]. Instead of me telling you everything upfront, ask me the questions you need to do this well. Ask one question at a time. When you have enough, produce [output]. // ExampleI want to build a coaching plan for an underperforming Product Consultant on my team. Ask me the questions you need to create a specific, actionable 30-day plan. Ask one at a time. // Model begins interviewingWhat is their current performance gap — and what metric best captures it? (e.g., demo-to-close rate, pipeline coverage, activity volume)
Best for: Discovery sessionsRequirements gatheringCoaching plansWhen you don't know what you don't know

Ask for Input Pattern

Instruct the model to prompt the user for information before proceeding

A lightweight version of Flipped Interaction. You tell the model to ask for specific inputs before it does anything. This is most useful in agent or tool contexts where you want the model to gatekeep itself — refusing to proceed until it has the data it needs to do the job correctly.

Ask for Input — FormatSystem
// In system prompt of an agentYou are a deal review assistant. Before analyzing any deal, always ask the user for: 1. Dealer name and market 2. Products being proposed 3. Deal stage and estimated close date 4. Known objections Do not proceed until all four are provided. // Vs. a one-shot user prompt versionBefore drafting my follow-up email, ask me for: the prospect's name, their main objection from last call, and my proposed next step. Then write the email.
Best for: Agents & toolsIntake formsPreventing hallucinationStructured workflows

Template Pattern

Give the model a fill-in-the-blank output structure and tell it to populate it

You define the exact output format — including placeholders — and instruct the model to fill them in. This is the most direct way to guarantee output structure. The model isn't deciding how to organize the response; it's populating a form you designed. Critical for any workflow that feeds model output into downstream systems.

AnalogyYou hand a contractor a document with fields already labeled and blanks waiting to be filled. They don't invent the structure — they complete yours. The result is always in the shape you need.
Template Pattern — Deal SummaryPattern
// PromptAnalyze the call transcript below and fill in this template exactly. Do not change the field names or add extra fields. TEMPLATE: Dealer Name: [DEALER_NAME] Products Discussed: [PRODUCT_LIST] Primary Objection: [OBJECTION] Objection Category: [TIMING | ROI | ADOPTION | PRICING | COMPETITION] Recommended Next Step: [NEXT_STEP] Deal Health: [RED | YELLOW | GREEN] Confidence Score: [1-10] // Transcript below...
Best for: CRM data extractionReport generationDownstream integrationsConsistent outputs

Recipe Pattern

Provide the goal and known constraints — ask the model to fill in the missing steps

You tell the model the desired end state, the ingredients you have (known steps or constraints), and any steps you want to lock in. It fills in the gaps to produce a complete, ordered sequence. It's CoT in reverse — you provide the destination and some waypoints, the model builds the full route.

AnalogyYou tell a chef: "I want a three-course meal, the main has to be salmon, and we start with soup. What goes between and how do I make it?" They fill in the missing steps around your constraints to produce a complete plan.
Recipe Pattern — Agent Rollout PlanPattern
// PromptI want to roll out our Email Triage Agent to the full MSS team of 50 PCs. I know the rollout must start with a pilot of 5 users and end with full adoption measurement at 90 days. Fill in a complete step-by-step rollout plan between those two points. Include any steps I may have missed. // Model produces Step 1: Select 5 pilot users (criteria: mix of tenure, product line, region) Step 2: Run 2-week pilot with structured feedback collection Step 3: Iterate on agent prompt based on pilot findings Step 4: Train next cohort of 15; document FAQs from pilot Step 5: Full rollout to remaining 30 with async training materials Step 6: 90-day adoption audit — usage rate, time saved, quality scores
Best for: Project planningProcess designRollout plansGap-filling workflows

Outline Expansion Pattern

Build large outputs incrementally — structure first, then expand section by section

Instead of asking the model to produce a long document in one shot (which degrades in quality), you first generate a high-level outline, then ask it to expand each section individually. The model stays focused, output quality stays high, and you can redirect at any outline node before investing in full generation.

AnalogyAn architect draws the floorplan before anyone starts framing walls. You review and approve the layout before a single nail goes in. Outline Expansion is the prompting equivalent — structure first, substance second.
Outline Expansion — Two-StepPattern
// Step 1 — Generate outlineCreate a high-level outline for a business case document proposing that Cox Automotive adopt an AI objection handler agent for the MSS team. Include 6–8 sections. Just the outline — no content yet. // Step 2 — Expand a specific sectionNow expand Section 3: "Current State Problem Analysis" into 3–4 paragraphs with specific data points and pain points from a 50-person product consultant team context. // Repeat for each section → complete document
Best for: Long documentsBusiness casesPlaybooksTraining content

Alternative Approaches Pattern

Ask the model to show you different ways to solve a problem, not just one answer

By default, a model will pick the most probable approach and run with it. The Alternative Approaches pattern instructs it to generate multiple distinct solutions — each with different tradeoffs — before you commit to one. This is essentially Tree of Thought made explicit as a user-facing instruction.

Alternative Approaches — FormatPattern
// TemplateWhenever I ask you how to do something, give me at least 3 alternative approaches. For each, explain the tradeoffs and which context it fits best. Then give your recommendation. // AppliedHow should I handle a dealer who says our pricing is too high? Approach A — Value anchor: Reframe around ROI, not cost. Best when dealer is data-driven. Approach B — Competitive compare: Compare TCO vs. alternatives. Best when competition is named. Approach C — Trial close: Ask "if price weren't the issue, would you move forward?" Best to isolate whether price is the real objection. Recommendation: Start with C to qualify, then use A or B.
Best for: Decision supportAvoiding tunnel visionSales strategyArchitecture choices

Fact Check List Pattern

Ask the model to flag its own uncertain claims after generating a response

After the model produces output, you instruct it to generate a list of every claim it made that might be inaccurate, outdated, or unverifiable — along with what you should check to confirm. It's a self-audit that surfaces the model's own uncertainty instead of hiding it behind confident prose.

AnalogyA good analyst doesn't just hand you a report — they attach a risk section that says "these three data points are based on 2023 estimates and should be verified before this goes to leadership." The Fact Check List pattern extracts that transparency automatically.
Fact Check List — FormatPattern
// Appended to any promptAfter your response, add a section called "Claims to Verify" that lists every statement I should independently confirm — including anything time-sensitive, product-specific, or based on assumptions you made about my context. // Model appends CLAIMS TO VERIFY: • "Industry average lead response time is 3.5 hours" — verify against current IHS Markit or CDK Global benchmark reports • "Xtime integration syncs in 90 seconds" — confirm with current release notes; this may have changed in recent updates • Assumed your team uses Salesforce CRM — confirm this is accurate
Best for: Research outputsCustomer-facing contentCompliance-sensitive docsExec presentations

Semantic Filter Pattern

Instruct the model to remove, transform, or redact content based on meaning — not just keywords

A semantic filter uses the model's language understanding to clean, redact, or transform content in ways a keyword search never could. You define what to remove or change by meaning, not by matching specific words. This is powerful for PII removal, tone normalization, audience-appropriate editing, and content moderation.

AnalogyA keyword filter strips "phone number" — but misses "reach me at 913-555-0100." A semantic filter understands that both are contact information and removes both. The difference is comprehension versus pattern matching.
Semantic Filter — ExamplesPattern
// PII RemovalRemove all personally identifiable information from the text below, including names, phone numbers, email addresses, company names, and any details that could identify a specific individual or organization. Replace each with [REDACTED]. // Tone FilterRewrite the following sales email to remove any language that could be perceived as high-pressure, manipulative, or creating false urgency. Keep the structure and facts — only change the emotional register. // Audience FilterRewrite the following technical documentation so that all jargon is either removed or explained in plain language suitable for a non-technical dealership manager.
Best for: PII/data redactionTone normalizationContent moderationAudience editing

Tail Generation Pattern

End every response with a prompt that keeps the conversation productively moving forward

Tail generation instructs the model to append a follow-up prompt, question, or call-to-action at the end of every response — maintaining conversational momentum. Instead of the model going silent after an answer, it actively suggests the next useful step. Especially valuable in interactive tools, coaching agents, and guided workflows.

Tail Generation — System Prompt ConfigSystem
// In system promptAt the end of every response, include one of the following: - A clarifying question if you need more information - A suggested next prompt the user might want to run - A recommended action step Label it clearly as "Next step:" or "You might also ask:" // Example tail appended to any response You might also ask: "Now generate a 30-day coaching plan based on the gap analysis above, focused specifically on discovery questioning."
Best for: Coaching agentsInteractive toolsGuided workflowsReducing blank-page friction
🎮

Game Play Pattern

Frame an interaction as a game with explicit rules to drive structured, engaging outputs

You define a game with rules, win conditions, and turns — and the model plays it. This isn't just for entertainment. Games are a powerful framing device for training scenarios, objection handling practice, decision simulations, and any situation where repeated structured engagement beats a one-shot answer.

AnalogyA flight simulator isn't a real plane — but the constraints of the simulation create genuine learning. Game Play prompting gives your AI interactions that same structured practice environment.
Game Play — Objection Handling TrainerSystem
// System prompt defining the gameWe are going to play an objection handling training game. RULES: 1. You play a skeptical dealership GM. I play the Product Consultant. 2. You will raise one objection per turn from this list: [pricing | timing | adoption | ROI | competitor] 3. After I respond, you score my answer 1–5 and explain why. 4. If I score below 3, you escalate the objection harder. 5. If I score 4–5 twice in a row, move to the next objection type. 6. After all 5 types, give me a final performance summary. START: Begin with a pricing objection. Set the scene first.
Best for: Sales trainingRoleplay scenariosDecision simulationsOnboarding tools
🔠

Meta Language Creation Pattern

Define a custom shorthand language the model interprets in a consistent, structured way

You teach the model a custom syntax — a personal shorthand or notation system — and it interprets it consistently throughout the conversation. This is how power users dramatically compress their prompts: instead of writing 40-word instructions every time, they write a 5-character command that the model already knows how to expand.

AnalogyA surgeon doesn't explain every instrument by its full name every time — the team knows that "10-blade" means a specific scalpel and acts immediately. Meta Language Creation is building that shared vocabulary with your AI.
Meta Language — Custom Prompt ShorthandSystem
// Define the meta language once in system promptI am going to define a shorthand language for our conversations. When I use these codes, interpret them exactly as defined: !OBJ [text] = Classify this objection and give a 3-part response !BRIEF [text]= Summarize in 3 bullets, max 15 words each !EXEC [text] = Rewrite for a C-suite audience, no jargon, under 100 words !COACH [text]= Give me direct coaching feedback on this as if you're my mentor !ALT [text] = Give me 3 alternative approaches to this // Now I can just type: !OBJ "We already use a competitor and my team won't switch."
Best for: Power usersRepeated workflowsLong sessionsCustom agent interfaces
📌

Menu Actions Pattern

Give the model a persistent menu of commands it executes on demand

You define a named set of actions in the system prompt — a menu — and the model executes whichever one you invoke by name. Unlike Meta Language (which is a syntax system), Menu Actions is a simple named-command interface. It's how you turn a general-purpose model into a specialized tool with consistent, repeatable behaviors your whole team can use.

Menu Actions — MSS Team AgentSystem
// Define the menu in system promptYou are the MSS Team AI Assistant. You support Product Consultants with deal prep, coaching, and communication. Available commands — execute exactly when invoked: DEAL REVIEW — Analyze a deal summary and give next-step recommendations OBJECTION PREP— Given a dealer profile, predict top 3 objections + responses EMAIL DRAFT — Draft a follow-up email from call notes I provide COACH ME — Give direct performance feedback on text I submit QUICK SUMMARY — Compress any input into 3 bullets under 15 words each HELP — List all available commands with descriptions To use, type the command name followed by your input.
Best for: Team-shared agentsConsistent toolingNon-technical usersCopilot agents

When to Use What

Why each pattern exists, when it earns its place, and when to skip it

Core Patterns
Zero-Shot Default
Why it exists: Most tasks don't need scaffolding. Zero-shot is the baseline — fast, simple, and sufficient for the majority of well-defined requests. It trusts that the model's training is enough.
✓The task is clear, common, and well-defined — summarize, translate, classify, explain
✓You're working with a capable frontier model (Claude, GPT-4, Gemini)
✓Speed matters more than perfection
✗The task requires a custom output format the model won't infer
✗Multi-step reasoning is involved — results will be unreliable
Few-Shot Show Don't Tell
Why it exists: Some patterns are easier to demonstrate than describe. Showing the model 3 examples of what "right" looks like is worth more than 300 words of instructions — especially for custom formats, classification schemes, or tones that are hard to articulate.
✓You have a custom output format, taxonomy, or style you can't easily describe
✓Zero-shot keeps getting the format wrong despite rephrasing
✓You're building a classification system with proprietary categories
✗You don't have good representative examples yet — bad examples hurt worse than none
✗The task is too variable for a fixed pattern to generalize
Chain-of-Thought Show Your Work
Why it exists: LLMs are next-token predictors — they don't naturally "pause to think." CoT forces intermediate reasoning steps into the token stream, which dramatically improves accuracy on problems where the path to the answer matters as much as the answer itself.
✓Multi-step math, logic, or analysis where errors compound
✓Decisions with dependencies — each step informs the next
✓You need to audit the reasoning, not just trust the conclusion
✗Simple factual lookups or one-step tasks — CoT adds tokens with no benefit
✗Output format is tightly constrained — verbose reasoning won't fit
ReAct Agent Engine
Why it exists: Static prompts can only work with what's already in the context window. ReAct breaks that ceiling — the model loops through Thought → Action → Observation, calling tools and updating its reasoning with real results. It's the only pattern that lets the model go get information rather than just recall it.
✓The answer requires live data — search, APIs, databases, knowledge bases
✓You're building an AI agent with tool access (Copilot Agent Builder, Agentforce)
✓Multi-step workflows where each step's output feeds the next decision
✗No tools are available — without actions, ReAct collapses into plain CoT
✗Latency is critical — each tool call adds round-trip time
Role / Persona Frame the Expert
Why it exists: The same question gets a very different answer depending on who you're asking. Role prompting activates domain-specific vocabulary, reasoning style, and risk tolerance without needing to specify every parameter explicitly. It's a cognitive shortcut that primes the model's entire response posture.
✓You need domain-specific framing — legal, financial, technical, sales, clinical
✓Adversarial review — "act as a skeptic and tear this apart"
✓Tone calibration — the persona carries the style implicitly
✗The task is purely mechanical — format extraction, data transformation
✗You need a fixed output schema — role affects tone, not structure
Structured (ICF) Production Workhorse
Why it exists: Ambiguity is the enemy of consistent output. Explicitly separating Instruction, Context, and Format removes every assumption the model would otherwise make. It's the most universally applicable pattern — and the foundation of any prompt that runs more than once.
✓Any prompt that will be reused, productized, or shared across a team
✓Output consistency matters — every run should produce the same shape
✓The context is rich and you want the model to use all of it correctly
✗Quick one-off questions — the structure adds overhead not worth paying
RAG Ground in Reality
Why it exists: LLM training data has a cutoff and no access to your proprietary knowledge. RAG bridges that gap by injecting relevant, current documents into the prompt at runtime. The model stops hallucinating details it doesn't know because the answer is right there in the context.
✓Answers must be grounded in your specific docs — product knowledge, policies, release notes
✓The knowledge base changes frequently — retraining isn't feasible
✓Accuracy and citation matter — support, compliance, customer-facing answers
✗Your knowledge base is small and stable — just paste the content directly
✗The question is general knowledge the model already knows well
Self-Consistency Trust the Consensus
Why it exists: A single reasoning path can contain silent errors. Running the same problem multiple times with varied reasoning and aggregating the results gives you a confidence signal — if most paths converge on the same answer, that answer is far more trustworthy than any single run.
✓The stakes of being wrong are high — a recommendation going to leadership
✓The problem is genuinely ambiguous and you want a confidence signal
✓You're comparing model judgments to human judgments for calibration
✗Cost and latency matter — you're paying for 3–5x the tokens
✗The task has a single correct answer — consensus doesn't improve factual recall
Tree of Thought Branch and Prune
Why it exists: For some problems, the first approach you think of isn't the best one. ToT forces deliberate exploration of multiple solution paths before committing — catching the "obvious but wrong" answer that a linear reasoning chain would have run with.
✓Strategic decisions with meaningfully different approaches worth comparing
✓Architecture or build decisions where early commitment is expensive to reverse
✓You suspect the obvious answer is wrong and want structured alternatives
✗One clear approach exists — branching creates noise, not insight
✗Speed is the priority — ToT is inherently verbose and slow
Interaction Patterns
Question Refinement Sharpen First
Why it exists: The question you ask is often not the question you need answered. Question Refinement surfaces that gap before the model wastes effort on the wrong target — especially valuable when you're exploring unfamiliar territory and don't know what you don't know.
✓You're in exploration mode — the problem space isn't fully defined yet
✓Previous answers felt off even when technically correct — your question was the problem
✓Building a coaching or strategy tool where question quality directly affects outcome
✗The question is already precise and operational — refinement adds friction
Cognitive Verifier Decompose First
Why it exists: Complex questions carry hidden sub-questions that need independent answers before a synthesis is possible. Jumping straight to a conclusion skips those sub-problems — producing confident answers built on shaky foundations. Cognitive Verifier makes the hidden structure visible.
✓The question sounds simple but actually contains 3–4 independent sub-problems
✓Build vs. buy, platform selection, strategic go/no-go decisions
✓You want to stress-test a recommendation before presenting it
✗The question is genuinely simple — decomposition creates false complexity
Audience Persona Calibrate the Reader
Why it exists: The same information lands differently depending on who's reading it. Without an audience signal, the model defaults to a generic technical-ish tone that fits no one perfectly. Audience Persona ensures the vocabulary, assumed knowledge, and framing match the actual reader — not an imaginary average.
✓Creating content for a specific role — GM, BDC rep, CFO, IT director
✓The same content needs multiple versions for different stakeholders
✓Training materials, onboarding docs, customer-facing explanations
✗Internal technical notes where audience is already implicit and consistent
Flipped Interaction Let It Ask You
Why it exists: You often don't know which information the model actually needs to help you well. Flipped Interaction inverts the burden — the model asks the right questions in the right order, gathering exactly what it needs. The result is far more tailored than anything you'd produce by front-loading everything upfront.
✓You know the desired output but not what inputs are required to get there
✓Discovery sessions, requirements gathering, coaching plans, intake flows
✓High-stakes personalized outputs — the more tailored, the more valuable
✗You're in a hurry — the dialogue loop takes time and patience
✗The task is well-specified — back-and-forth adds no value
Ask for Input Gate Before Acting
Why it exists: Agents and automated tools that proceed without sufficient context produce confidently wrong results. Ask for Input is the guardrail — it tells the model to stop and collect what it needs before doing anything. It prevents the model from filling gaps with assumptions.
✓Agent system prompts where missing context leads to hallucinated specifics
✓Intake flows — deal review, coaching requests, ticket triage
✓Any workflow where the user typically forgets to include key details
✗Contexts where the user has already provided complete information
Output & Format Patterns
Template Pattern Lock the Shape
Why it exists: When output feeds a downstream system — a CRM field, a report generator, a dashboard — format variability is a breaking failure, not just an aesthetic preference. Template Pattern eliminates that variability entirely by giving the model a literal form to fill in rather than a description of what to produce.
✓Output feeds a CRM, database, API, or structured workflow
✓Multiple people need to produce consistently formatted outputs
✓You're extracting structured data from unstructured input (call transcripts, emails)
✗The output is inherently narrative — forcing it into a template loses meaning
Recipe Pattern Fill the Gaps
Why it exists: You often know where you need to start and where you need to end up, but not every step in between. Recipe Pattern lets you lock down the non-negotiables and ask the model to complete the path — while flagging steps you may have missed. It's collaborative planning rather than delegation.
✓Project plans where you have hard constraints at both ends but gaps in the middle
✓Rollout plans, implementation sequences, change management workflows
✓Process design where you know the inputs and outputs but not the full procedure
✗Tasks with no meaningful constraints — the model needs some anchor points to work from
Outline Expansion Structure Then Fill
Why it exists: Long-form generation degrades in quality as length increases — the model loses coherence, repeats itself, or drifts from the original intent. Outline Expansion solves this by separating structure decisions from content decisions. You review and approve the architecture before any content is committed.
✓Any document over ~500 words — business cases, playbooks, training guides
✓You want creative control over structure before investing in generation
✓Multiple stakeholders need to review and redirect before content is written
✗Short outputs — the two-step adds overhead that isn't worth it
Alternative Approaches Escape the Default
Why it exists: Models have strong priors — they'll default to the most common or obvious solution unless you explicitly ask for alternatives. This is fine when the obvious answer is correct. It's a problem when the obvious answer is what everyone else is already doing. Alternative Approaches forces the model out of its default.
✓Strategic decisions where the first answer is probably conventional wisdom
✓You want tradeoff visibility before committing to an approach
✓Brainstorming sessions, option analysis, competitive positioning
✗There is genuinely only one right approach — alternatives create false equivalence
Fact Check List Surface the Risk
Why it exists: LLMs present uncertain information with the same confident tone as verified facts. The Fact Check List pattern breaks that illusion — asking the model to explicitly flag which of its own claims you should verify before trusting. It converts invisible risk into a visible checklist.
✓Output is going to a customer, executive, or external stakeholder
✓The topic involves statistics, product specs, pricing, or compliance details
✓You're doing research synthesis and need to know what to verify
✗Internal brainstorming where perfect accuracy isn't the goal
Semantic Filter Clean by Meaning
Why it exists: Keyword-based filtering is brittle — it misses synonyms, context-dependent phrasing, and implied meaning. Semantic Filter uses the model's actual language understanding to remove or transform content based on what it means, not what words it uses. That's a capability no regex or rule engine can match.
✓PII redaction where contact info appears in many unpredictable forms
✓Tone normalization — remove aggressive, passive-aggressive, or high-pressure language
✓Audience-appropriate editing — strip jargon, simplify for a specific reader
✗Simple word replacement where a find-and-replace is faster and more reliable
Tail Generation Keep Moving
Why it exists: Most users stall after receiving an answer — not because they're done, but because they don't know what to ask next. Tail Generation removes that friction by surfacing the next logical step automatically. In agent and tool contexts it's the difference between a dead-end and a guided experience.
✓Interactive tools, coaching agents, and guided workflows where momentum matters
✓Users who are non-technical and unlikely to know what to ask next
✓Onboarding experiences where you want users to discover more capabilities
✗Power users who find the suggestions patronizing or in the way
✗One-shot tasks where no follow-up is expected or needed
Advanced Patterns
Game Play Practice Arena
Why it exists: Structured constraints drive better learning and engagement than open-ended interaction. A game provides rules, turns, scoring, and stakes — which activate a different cognitive mode than a standard Q&A. For training scenarios especially, the game framing turns passive reading into active practice.
✓Sales training — objection handling, discovery questioning, closing practice
✓Decision simulations where you want to practice judgment, not just read about it
✓Onboarding scenarios that benefit from interactive challenge over static content
✗Analytical tasks — game framing adds noise without benefit
✗Time-sensitive work environments where the game structure feels frivolous
Meta Language Build Your DSL
Why it exists: Repeated prompting creates repeated overhead — the same 40-word instruction written dozens of times per session. Meta Language creation lets you define a compact shorthand once and reuse it indefinitely. It's the difference between typing a full SQL query every time and having stored procedures.
✓Power users who run the same types of prompts repeatedly across long sessions
✓Building internal tools where your team needs consistent commands
✓Agent interfaces where users interact via structured inputs rather than freeform text
✗Occasional users who won't remember the syntax — it creates more confusion than efficiency
✗Single-session tasks — the setup cost exceeds the benefit
Menu Actions Named Commands
Why it exists: Not everyone using an AI tool knows how to prompt well. Menu Actions gives them a curated set of named commands that always produce consistent, well-designed outputs — without requiring prompting knowledge. It's the UX layer that makes an LLM approachable for a whole team, not just the people who built it.
✓Team-shared agents where users have varying AI experience
✓Copilot Agent Builder or any interface where you control the system prompt
✓Workflows with a defined, repeatable set of actions — not open exploration
✗Open-ended research or creative tasks that don't fit into predefined buckets
✗Expert users who benefit more from freeform access than a constrained menu
The layering principle: These patterns aren't mutually exclusive — the best production prompts stack several at once. A customer-facing agent might combine a System Prompt (identity + rules) + Menu Actions (user interface) + RAG (live knowledge) + Template (output shape) + Tail Generation (momentum). Each pattern solves a different failure mode — together they close almost every gap.
