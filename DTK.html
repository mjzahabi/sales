Section 01 · VS Code

What is VS Code — and why use it?

🌱 Beginner

VS Code (Visual Studio Code) is a free text editor made by Microsoft. But calling it a "text editor" is like calling a Swiss Army knife a "small knife" — technically true, but it misses the point. It's a full-featured workspace for managing any kind of structured text files: code, system prompts, documentation, HTML guides, markdown notes, JSON configs — all in one place.

📖 Analogy

Think of VS Code the way you'd think of Microsoft Word, except instead of being built for polished documents with fonts and formatting, it's built for working files — files you edit, organize, reference, and iterate on. It's your workbench, not your publishing tool.

The reason it matters for your work specifically: right now when you write a system prompt, it lives inside a chat window. VS Code gives that same prompt a real home — with a filename, a folder structure, syntax highlighting, and a history of every change you've ever made. It turns your prompts from disposable chat messages into managed, organized assets.

💡
For non-coders specifically:

You don't need to write a single line of code to get enormous value from VS Code. The file explorer, search, and split-view features alone make it worth using just for prompt management.

⚡ Advanced

VS Code is built on Electron and uses the Monaco editor engine — the same engine that powers GitHub's web-based editor. It supports Language Server Protocol (LSP), meaning any language with an LSP implementation gets full IDE features: intellisense, go-to-definition, inline errors. For your AI tooling work, the most relevant advanced capabilities are the integrated terminal (full shell access without leaving the editor), multi-root workspaces (multiple repo folders in one window), and the remote development extensions for working on containers or remote servers.

What VS Code does that a chat window can't

🗂️

Organizes everything in one place

All your agents, prompts, reference docs, and HTML guides live in one folder you can see, search, and navigate instantly.

↔️

Side-by-side comparison

Open two versions of a system prompt next to each other and see exactly what changed between v1 and v3.

🔍

Search across everything

One keyboard shortcut searches every word in every file in your entire workspace — find any guardrail, persona definition, or note instantly.

🔌

Extensible with add-ons

Extensions add capabilities — Markdown preview, Git history, word count, syntax highlighting for any file type.

Where to get it, how to install it

It's free. VS Code is made by Microsoft and costs nothing. There's no sign-up required to use it — just download and install.

1

Go to the official download page

Open your browser and go to code.visualstudio.com. The site will automatically detect your operating system (Windows or Mac) and show the right download button.

💡
Only download from code.visualstudio.com

The official Microsoft site is the only safe source. Avoid downloading from third-party sites — they may bundle unwanted software.

2

On Windows: click the big blue Download button. This downloads a .exe file. Open it, click through the installer accepting the defaults — no special options are needed. VS Code will appear in your Start menu when finished.

When VS Code opens you'll see a Welcome tab. You can close this. The most important thing on first launch: go to File → Open Folder and open a folder you want to work in. VS Code works with folders, not just individual files — a folder is your workspace.

ℹ️
Windows users: add "Open with Code" to right-click menu

During installation, check the box that says "Add 'Open with Code' action to Windows Explorer file context menu." This lets you right-click any folder on your computer and open it in VS Code instantly — a huge time-saver.

Section 03 · VS Code

Working with files and folders

VS Code is built around the concept of a workspace — a single folder that you open and work inside. Everything you do in VS Code happens relative to that folder. Think of it like opening a project binder: once it's open, you have access to everything inside.

Opening and creating your workspace

1

Create a folder on your computer first

Before opening VS Code, create a regular folder somewhere sensible — like your Documents folder. Name it something like claude-projects. This will become your workspace root.

2

Open that folder in VS Code

Go to File → Open Folder (Mac: File → Open) and navigate to the folder you just created. Click Select Folder (or Open on Mac). VS Code now shows that folder's contents in the Explorer panel.

3

Create a new file

In the Explorer panel, hover over your folder name and click the New File icon (looks like a page with a +). Type your filename and hit Enter. For a system prompt, name it something like onboarding-agent.md — the .md extension means Markdown, which gives you readable formatting.

💡
File extensions matter

The extension tells VS Code (and other tools) what kind of file it is. Use .md for notes and prompts (Markdown), .html for web pages, .txt for plain text, .json for structured data.

4

Create subfolders for organization

Hover over your folder name and click the New Folder icon. Create a structure that matches your work — separate folders per agent, a reference folder, a guides folder. You can drag and drop files to reorganize anytime.

5

Rename, move, and delete files

Right-click any file or folder in the Explorer panel to get options: Rename, Move, Delete, Reveal in Explorer (opens the actual folder on your desktop). You can also drag files between folders directly in the panel.

Editing files — the features you'll use daily

Split view — your most powerful feature

Split view lets you see two files side by side. The single best use for your work: open system-prompt-v2.md and system-prompt-v3.md simultaneously to compare what changed.

1

Open the first file

Click any file in the Explorer to open it in the editor.

2

Split the editor

Press Ctrl/Cmd + \ to split the editor into two columns. Or right-click a file tab and select Split Right.

3

Open the second file in the new column

Click on the right-side panel to focus it, then click your second file from the Explorer. Both files are now visible simultaneously. You can edit either one.

Markdown preview — see your prompts rendered

When you're working with .md files, VS Code can show you a live rendered preview so you see headings, bullet points, and bold text properly — not just raw symbols.

With a Markdown file open, press Ctrl/Cmd + Shift + V to open a preview panel, or click the preview icon in the top-right corner of the editor (looks like a split page with a magnifying glass).

Find and replace across all files

One of VS Code's most useful features for maintaining a prompt library. Need to update your company name everywhere? Change a guardrail phrase across all agents? One search finds and replaces it all.

Section 05 · GitHub

What is GitHub — and what's Git?

🌱 Beginner

There are two things here that often get confused: Git and GitHub. They're related but different.

⚙️

Git — the engine

Git is software that tracks every change made to every file in a folder. It runs on your computer. Think of it as the "track changes" feature from Word, except for any file, and with a permanent, detailed history.

☁️

GitHub — the cloud layer

GitHub is a website that stores your Git-tracked folders online. It's the backup, the history viewer, and the collaboration layer. Git does the tracking locally; GitHub keeps it safe in the cloud.

📖 Analogy

Git is like a personal diary that records every edit you make to a document with timestamps and notes. GitHub is like a safe deposit box where you store that diary — accessible from anywhere, never lost, shareable with others if you choose.

Why version control changes how you work

Without version control, your workflow probably looks like this: edit a system prompt, it stops working well, you try to remember what you changed, maybe you have a system-prompt-backup.txt somewhere that might be the right version... With Git and GitHub, every version you've ever saved is permanently stored with a note explaining what changed and why. Going back is one click.

ℹ️
GitHub ≠ Google Drive

Google Drive autosaves and syncs files continuously. GitHub is intentional — you choose when to save a "snapshot" (called a commit) and write a message explaining what that snapshot represents. This makes the history meaningful rather than just a timestamp list.

⚡ Advanced

Git is a distributed version control system — every clone of a repo contains the full history, not just the latest snapshot. Commits are identified by SHA-1 hashes, and the DAG (directed acyclic graph) of commits is what makes branching, rebasing, and cherry-picking possible. GitHub adds a collaboration layer on top: pull requests with code review, protected branches, Actions for CI/CD, and the API surface for automating workflows. For AI tooling, the relevant GitHub features are: Actions (auto-run tests or deploy on push), Secrets (secure env var storage at the repo level), and Packages (host private npm packages for shared internal libraries).

Section 06 · GitHub

The five concepts you must understand

GitHub has a lot of vocabulary. Here are the five terms that actually matter for your use case, explained plainly.

REPO

Repository (repo)

A repository is simply a folder that Git is tracking. Your claude-agent-prompts folder becomes a repository once you initialize Git inside it. Everything inside that folder — every file, every subfolder — is tracked. One project = one repo is a good default rule.

CMMT

Commit

A commit is a saved snapshot of your files at a specific moment, with a message you write explaining what changed. Think of each commit like a chapter in a book — it has a title (your commit message) and represents a meaningful unit of progress. You decide when to commit, unlike autosave. A good rule: commit whenever you finish a meaningful change.

Good commit message examples

"Add escalation guardrail for out-of-scope questions"
"Revise persona — too informal, now role-specific"
"v3 system prompt — split into modular sections"

PUSH

Push

A push sends your local commits up to GitHub. You can make many commits locally and then push them all at once — or push after every commit. Either way, nothing is on GitHub until you push. Commit = save locally. Push = send to cloud.

BRNCH

Branch

A branch is a parallel copy of your files where you can experiment freely without affecting your main version. Imagine forking a timeline in a movie — you go off and test things in a separate timeline, then either merge the results back or discard them. Your main timeline (called main) stays untouched until you're ready.

💡
When to branch:

Create a branch whenever you want to test a significant prompt change without risking your working version. Name it something descriptive: experiment/add-rag-context or test/tone-shift.

PULL

Pull

A pull brings changes from GitHub down to your local computer. If you work on multiple devices, or if a collaborator has pushed changes, you pull to get the latest version before you start working. It's the mirror of push: push goes up, pull comes down.

Section 08 · GitHub Desktop

How to use GitHub Desktop — every panel explained

GitHub Desktop has three main views. Here's exactly what each one is for.

The Changes tab — your staging area

This is the first thing you see when you open GitHub Desktop. Any file you've edited since your last commit appears here with a checkbox. The right side shows a diff — a color-coded view of exactly what changed: green lines were added, red lines were removed.

💡
Always review the diff before committing

Click any file in the Changes tab and look at the diff on the right. This is your chance to verify you're committing exactly what you intend to — and nothing accidental. This habit catches mistakes before they're in your permanent history.

How to make a commit

1

Check the files you want to include

In the Changes tab, each changed file has a checkbox. By default all changes are checked. If you want to commit some files but not others, uncheck the ones you want to leave out.

2

Write a commit message

At the bottom-left of GitHub Desktop is a text box with "Summary (required)." Write a short, specific description of what this change does. Keep it under 72 characters. The optional Description field is for longer notes if you need them.

3

Click "Commit to main"

This saves the snapshot locally. The files disappear from the Changes tab because there's nothing new to commit.

4

Click "Push origin"

The button in the top bar changes to say "Push origin" (or shows an arrow icon). Click it. Your commit is now on GitHub. That's the full cycle.

The History tab — your time machine

Click the History tab to see every commit ever made to this repo, in reverse chronological order. Click any commit to see exactly what files changed and what the diff was. To restore an older version: right-click a commit and select Revert Changes in Commit — this creates a new commit that undoes that one, rather than erasing history.

Working with branches

1

Create a new branch

In GitHub Desktop, click the Current Branch dropdown at the top center. Click New Branch. Give it a name that describes the experiment: experiment/tone-shift or test/new-guardrails. Click Create Branch.

You're now working on a separate copy. Any commits you make only affect this branch.

2

Make and commit your experimental changes

Edit your files in VS Code, commit in GitHub Desktop as normal. These commits live only in your branch.

3

Switch back to main anytime

Click the Current Branch dropdown and select main. Your files instantly revert to the main version. Your experimental changes are safely stored in the branch, not lost — just not visible until you switch back.

4

Merge or discard the branch

If the experiment worked: in GitHub Desktop, switch to main, then go to Branch → Merge into Current Branch, select your experiment branch. Your changes are now merged into main.

If the experiment failed: switch to main, go to Branch → Delete, select the branch. It's gone. Main is untouched.

Before you close VS Code, push your commits to GitHub. One push can contain multiple commits. Your work is now safely backed up in the cloud and visible in your repo history.

Section 10 · Going Deeper

Next-level features worth learning

GitHub Pages — publish your HTML guides

Any repo can become a live website for free with GitHub Pages. Once enabled, your HTML files get a permanent public URL automatically updated every time you push.

1

Put your HTML file in the repo root

For the main URL to load your guide, name it index.html and place it in the root of your repo (not inside a subfolder). Other HTML files can live in subfolders and are accessible at their path.

2

Enable GitHub Pages on github.com

Go to your repo on github.com → Settings → Pages (left sidebar) → Under Source, select Deploy from a branch → Choose main and / (root) → Save.

3

Your site is live

Within a minute, your site is at yourusername.github.io/repo-name. Every push automatically updates it. Share the URL anywhere — no login required to view it.

GitLens extension — Git history inside VS Code

Install the GitLens extension and Git history becomes visible right inside your editor. You'll see who last changed each line, when, and what the commit message was — all without leaving VS Code or opening GitHub Desktop.

Using GitHub.com to compare changes

On github.com, navigate to any file and click History to see all commits that touched that file. Click any commit to see a beautiful diff view. Section 12 · React

What is React — and why does it matter?

🌱 Beginner

React is a JavaScript library created by Meta (Facebook) for building user interfaces. Where a regular HTML page is static — you write it once and it looks the same to everyone — a React app is dynamic and interactive. It can update, respond to clicks, load new data, and change what's on screen without ever reloading the page.

📖 Analogy

Think of a traditional HTML page like a printed flyer — once it's printed, it's fixed. React is more like a digital display board: the structure is defined once, but the content can change in real time based on what's happening. A button click changes what's shown. New data arrives and the screen updates automatically.

React is the most widely used frontend framework in the world. When you use Claude's artifact environment and build something with interactive buttons, live state, or dynamic data — that's React. When Anthropic, Netflix, Airbnb, and thousands of other companies build their web interfaces, most of them use React.

ℹ️
The Claude connection

When Claude builds you a .jsx artifact — the interactive cards, live counters, filterable lists — that's React. You've been using React-rendered outputs already. Learning React means you can understand, modify, and extend those outputs rather than treating them as a black box.

⚡ Advanced

React uses a virtual DOM to batch and minimize actual DOM mutations — when state changes, React diffs the previous and next virtual DOM trees and applies only the minimal set of real DOM updates. The Fiber architecture (introduced in React 16) enables interruptible rendering, concurrent features, and Suspense for async data loading. For your AI tooling work, the most relevant patterns are: controlled components for form state, useCallback/useMemo for preventing unnecessary re-renders in data-heavy dashboards, and Context API or Zustand for sharing agent state across components without prop drilling. Consider React Query (TanStack) for managing Claude API call state — it handles loading, error, caching, and refetching out of the box.

What makes React different from plain HTML

🧱

Components — reusable building blocks

Instead of writing the same button HTML five times, you define a Button component once and drop it anywhere. Change it once, it updates everywhere it's used.

🔁

State — memory that drives the UI

React components remember things — like whether a menu is open, how many items are in a cart, or what a user typed. When that memory (state) changes, the UI updates automatically.

⚡

Fast updates without page reloads

React only updates the exact parts of the page that changed — not the whole thing. This makes apps feel instant and fluid rather than clunky.

🌐

Huge ecosystem

Thousands of pre-built component libraries, tools, and frameworks (like Next.js) are built on top of React. Whatever you want to build, someone's already made a starting point.

It's a document, guide, or reference pageIt's an app — a tool, dashboard, or interactive experience
No data fetching or API calls neededYou need to load data from an external source
One person builds it and it stays the sameMultiple components reuse the same UI patterns
Speed of creation matters more than complexityLong-term maintainability matters

Section 13 · React

Core concepts you need to understand

React has its own vocabulary. These are the six concepts that actually matter — everything else is built on top of them.

COMP

Components

A component is a self-contained piece of UI — a button, a card, a navigation bar, a whole page. It's written as a JavaScript function that returns HTML-like syntax (called JSX). Components can contain other components, like Russian nesting dolls.

WelcomeCard.jsx — a simple component 
// A component is just a function that returns UI
function WelcomeCard({ name }) {
  return (
    <div className="card">
      <h2>Hello, {name}!</h2>
      <p>Welcome to the dashboard.</p>
    </div>
  );
}

// Use it anywhere like an HTML tag:
<WelcomeCard name="Mo" />
PROPS

Props — passing data into components

Props (short for properties) are how you pass data into a component from the outside. In the example above, name="Mo" is a prop. Think of them like the settings or inputs for a component — they customize how it looks and behaves without changing the component itself.

💡
Props flow one direction — down.

A parent component passes props to a child component. The child can read them but can't directly change them. This one-way flow makes React apps predictable and easier to reason about.

STATE

State — a component's memory

State is data that lives inside a component and can change over time. When state changes, React automatically re-renders that component with the new value. The primary tool for state is the useState hook.

Counter.jsx — state in action 
import { useState } from 'react';

function Counter() {
  // count is the value. setCount updates it.
  const [count, setCount] = useState(0);

  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={() => setCount(count + 1)}>
        Add one
      </button>
    </div>
  );
}
// Every click updates count → UI re-renders automatically
JSX

JSX — HTML inside JavaScript

JSX is the HTML-like syntax you write inside React components. It looks like HTML but it's actually JavaScript — which means you can embed expressions inside curly braces {}. className is used instead of class (because class is a reserved word in JavaScript).

⚠️
JSX ≠ HTML

It looks almost identical but there are small differences. Every tag must be closed (<br /> not <br>). Use className not class. Use onClick not onclick. Your browser won't run JSX directly — it gets compiled to regular JavaScript first.

HOOK

Hooks — superpowers for components

Hooks are built-in React functions that add capabilities to your components. You've seen useState for managing state. The other one you'll use most is useEffect — it runs code when a component loads or when specific values change (like fetching data when a page opens).

HookWhat it doesWhen to use it
useStateStores and updates a value in a componentAny time the UI needs to remember something
useEffectRuns code after render, or when values changeFetching data, subscriptions, timers
useRefStores a value that doesn't trigger re-rendersAccessing DOM elements directly
useMemoCaches an expensive calculationPerformance optimization
FILE

File structure — how React projects are organized

A React project created with the standard tools has a predictable structure. The key folder is src/ — that's where all your components live. public/ holds static assets. package.json lists all your dependencies.

📁 my-react-app/ ├── 📁 src/ ← your code lives here │ ├── App.jsx ← root component, entry point │ ├── main.jsx ← mounts App into the HTML page │ └── 📁 components/ ← your reusable components │ ├── Header.jsx │ └── Card.jsx ├── 📁 public/ ← images, icons, static files ├── index.html ← the single HTML page React mounts into ├── package.json ← project config and dependencies └── vite.config.js ← build tool configuration

Section 24 · Postman

What is Postman — and why use it?

🌱 Beginner

Postman is a tool for testing APIs. An API (Application Programming Interface) is how software talks to other software — when your React app calls Claude, it's making an API request. Postman lets you send those requests manually, outside of any code, and inspect exactly what comes back.

📖 Analogy

If an API is a restaurant kitchen and your code is a waiter that places orders, Postman is like walking directly into the kitchen yourself and placing the order at the pass — you see exactly what comes out, with no middleman. When something is wrong with your order, you know whether the problem is in the kitchen (the API) or with the waiter (your code).

Why Postman matters for your AI work

When you build something that calls the Claude API and it doesn't work, there are two possible failure points: your code has a bug, or the API call itself is wrong. Postman isolates the API call completely — if it works in Postman but not in your code, you know the bug is in the code. If it fails in Postman too, the problem is the API call — wrong parameters, wrong API key, wrong format.

⚡ Advanced

Postman supports pre-request scripts and test scripts in JavaScript — you can programmatically set headers, chain requests (use the response from one as input to the next), and write assertions that run automatically after each call. For Claude API testing, the critical advanced pattern is Collection Runner: define a data file of different system prompts, run them all in sequence automatically, and export the results. This is how you regression test your agents — run 50 test cases against a new system prompt version and flag any that changed significantly. Also worth knowing: Postman's Mock Server feature lets you stub the Claude API with fixed responses during frontend development, so you don't burn real API credits while testing UI states.

🔍

Debug API calls in isolation

Test your Claude API call directly — before writing a single line of React code. See the full response, headers, status codes, and error messages clearly.

💾

Save and organize requests

Build a library of saved API calls — different system prompts, different models, different parameters. Run any of them with one click whenever you need to test something.

📊

See the full raw response

Claude's API returns JSON with specific fields. Postman shows you the complete raw response so you know exactly how to reference the data you need in your code.

🔄

Iterate on prompts fast

Change a system prompt, hit Send, see the response. No rebuilding your React app. No reloading. Postman is the fastest feedback loop for prompt testing outside of Claude.ai itself.

What is Supabase — and why use it?

Supabase is a free, hosted database with an API built in. It gives your React apps the ability to store, retrieve, and update data — without you needing to build or manage a server. Think of it as a spreadsheet your app can read and write to programmatically, in real time.

🌱 Beginner
📖 Analogy

⚡ Advanced

Supabase is a Firebase alternative built on open-source PostgreSQL rather than a proprietary NoSQL store. The auto-generated REST and GraphQL APIs are built on PostgREST — they respect PostgreSQL's Row Level Security (RLS) policies, which means your security rules live in the database layer, not in your application code. For AI tooling, this matters: you can write an RLS policy that ensures each user only sees their own prompt logs without touching a line of JavaScript. The Supabase Edge Functions (Deno-based) let you run server-side logic close to the database — useful for post-processing Claude responses before storing them, or for triggering actions when new data arrives.

Supabase uses PostgreSQL — the same database technology used by serious enterprise applications. Your data is structured in tables with rows and columns, like a spreadsheet but far more powerful.

⚡

Instant API — no backend code needed

Supabase auto-generates an API for your tables the moment you create them. Your React app can query, insert, update, and delete data with a few lines of JavaScript — no server to build.

🔒

Authentication built in

Supabase handles user login — email/password, Google, GitHub, and more. Add a login system to your React app without building auth from scratch.

Real-time updates

Building your first Chrome Extension

🌱 Beginner

A Chrome Extension is just a folder of files — no special tools required to get started. The most important file is manifest.json, which is a settings file that tells Chrome what your extension is, what permissions it needs, and which files do what.

The anatomy of an extension

📁 my-extension/ — required files
manifest.json

The control file. Declares the extension's name, version, permissions, and which scripts run where. Required — every extension needs this.

popup.html

The UI that appears when you click the extension icon in the toolbar. Built in regular HTML — can include React if needed.

content.js

Runs inside the web page the user is viewing. Can read the page DOM, extract data, and inject new UI elements directly into the page.

background.js

A service worker that runs in the background. Handles API calls (like Claude), manages state, and coordinates between popup and content scripts.

icons/

PNG icons in various sizes (16, 48, 128px). Shown in the toolbar, extensions page, and Chrome Web Store.

styles.css

Optional. Styles for your popup UI or injected content. Scoped to your extension — doesn't affect the host page's styles.

Step-by-step: build a Claude-powered extension

1

Create the project folder

In VS Code, create a new folder called claude-helper-extension. This is your extension's home — all files go here. No npm, no Node.js required for a basic extension.

2

Create manifest.json

This is the most important file — Chrome reads it first to understand your extension.

manifest.json 
{
  "manifest_version": 3,
  "name": "Claude Page Helper",
  "version": "1.0",
  "description": "Summarize any page with Claude",
  "permissions": ["activeTab", "storage"],
  "host_permissions": ["https://api.anthropic.com/*"],
  "action": {
    "default_popup": "popup.html",
    "default_icon": { "48": "icons/icon48.png" }
  },
  "content_scripts": [{
    "matches": ["<all_urls>"],
    "js": ["content.js"]
  }],
  "background": {
    "service_worker": "background.js"
  }
}
3

Create popup.html — your toolbar UI

popup.html 
<!DOCTYPE html>
<html>
<head>
  <style>
    body { width: 320px; padding: 16px; font-family: sans-serif; }
    button { width: 100%; padding: 10px; background: #1a6fff;
             color: white; border: none; border-radius: 6px; cursor: pointer; }
    #result { margin-top: 12px; font-size: 13px; line-height: 1.6; }
  </style>
</head>
<body>
  <button id="summarize">✨ Summarize this page</button>
  <div id="result"></div>
  <script src="popup.js"></script>
</body>
</html>
4

Create popup.js — connect button to Claude

popup.js 
document.getElementById('summarize').addEventListener('click', async () => {
  // 1. Get the current tab
  const [tab] = await chrome.tabs.query({ active: true, currentWindow: true });

  // 2. Ask content.js to extract the page text
  const response = await chrome.tabs.sendMessage(tab.id, { action: 'getPageText' });

  // 3. Send to Claude API
  const result = await fetch('https://api.anthropic.com/v1/messages', {
    method: 'POST',
    headers: {
      'x-api-key': 'YOUR_API_KEY',
      'anthropic-version': '2023-06-01',
      'content-type': 'application/json'
    },
    body: JSON.stringify({
      model: 'claude-sonnet-4-6',
      max_tokens: 512,
      messages: [{ role: 'user', content: `Summarize this page in 3 bullet points:\n\n${response.text}` }]
    })
  });

  const data = await result.json();
  document.getElementById('result').innerText = data.content[0].text;
});

Hardcoding your API key in popup.js is fine for local testing but not for distributing the extension. For production, use one of these patterns:

Option 1 — User-provided key: Add an options page where users paste their own Anthropic API key. Store it with chrome.storage.sync. The key stays in the user's browser, not your code.

Option 2 — Proxy server: Build a small Vercel serverless function that holds the key server-side. Your extension calls your Vercel URL, which calls Claude. The key never touches the extension at all.

Option 2 is the right approach for internal team tools — you control the key, and you can revoke it without pushing a new extension version.

5

Create content.js — read the page

content.js 
// Listen for messages from popup.js
chrome.runtime.onMessage.addListener((request, sender, sendResponse) => {
  if (request.action === 'getPageText') {
    // Extract visible text from the page
    const text = document.body.innerText.slice(0, 4000); // limit length
    sendResponse({ text });
  }
  return true; // keeps message channel open for async
});
6

Load it in Chrome — no publishing required

To test your extension locally:

a

Open Chrome and go to chrome://extensions

b

Toggle Developer mode on (top right)

c

Click Load unpacked and select your extension folder

d

Your extension appears in the toolbar. Click its icon to open the popup. After any code change, click the refresh icon on the extension card to reload it.

Commit your extension to GitHub

Your extension folder is just files — version control it exactly like any other project. Push to a private GitHub repo, share the repo with teammates, they load it the same way. No Chrome Web Store publishing needed for internal tools.

Reference

Glossary — plain-English definitions

Every term you'll encounter in this guide and in day-to-day AI development work, explained without jargon.

API General

Application Programming Interface. A set of rules that lets two pieces of software talk to each other. When your app calls Claude, it's using Anthropic's API — sending a structured request and receiving a structured response.

Repository (Repo) GitHub

A folder whose contents Git is tracking. Think of it as a project folder with a full memory — every file change ever made is recorded.

Commit GitHub

A saved snapshot of your files at a specific moment, with a message you write. Like a checkpoint in a video game — you can always return to it.

Push / Pull GitHub

Push sends your local commits up to GitHub. Pull brings changes from GitHub down to your computer. Push = upload. Pull = download.

Branch GitHub

A parallel copy of your files where you can experiment without affecting the main version. If the experiment works, merge it in. If not, delete the branch.

Pull Request (PR) GitHub

A proposal to merge changes from one branch into another. On GitHub.com, a PR shows a diff of every change and can require review before merging.

Component React

A self-contained, reusable piece of UI — a button, a card, a form. Defined once as a function, used anywhere. Changing one component updates every place it's used.

State React

A component's internal memory — values it keeps track of that can change over time. When state changes, React automatically updates the UI to reflect the new value.

Props React

Data passed into a component from outside, like function arguments. Props let you customize a component's appearance or behavior without changing its code.

Hook React

A built-in React function that adds capabilities to components. useState adds memory. useEffect runs code when the component loads or values change.

JSX React

The HTML-like syntax used inside React components. Looks like HTML but is JavaScript — you can embed expressions in curly braces {}. Compiled to regular JS before the browser runs it.

npm React / Node

Node Package Manager. A tool that installs libraries (packages) your project needs. Running npm install react downloads React into your project folder.

Node.js React / General

JavaScript's runtime engine for your computer. Normally JavaScript runs inside a browser — Node.js lets it run outside the browser, on your machine or a server.

Vite React

The modern tool for creating and running React projects. It sets up the project structure, runs a local dev server, and compiles your code for deployment. Replaced Create React App.

Deploy Vercel

Publishing your app to a live server so others can access it. Vercel automates this — every GitHub push triggers a new deployment automatically.

Environment Variable Vercel / General

A configuration value stored outside your code — typically for secrets like API keys. In Vercel, set them in the dashboard so they're available to your app without being in your codebase.

Serverless Function Vercel

Backend code that runs on demand without a dedicated server. In Vercel, you write a function file and it becomes an API endpoint — runs only when called, costs nothing when idle.

localhost General

Your own computer's local server address. When npm run dev starts, it serves your app at localhost:5173 — only you can see it, not the internet.

JSON General

JavaScript Object Notation. The standard format for sending data between apps. Looks like a structured list of key-value pairs. Claude's API returns responses in JSON.

REST API / Endpoint General

A URL you can call to interact with a service. https://api.anthropic.com/v1/messages is Claude's endpoint. You send a POST request to it; it returns Claude's response.

Database Supabase

Organized, persistent storage for structured data. Tables hold rows of records (like spreadsheet rows). Supabase provides a hosted database your app can read and write to instantly.

Query Supabase

A request to a database for specific data. "Give me all rows where status = 'active'" is a query. Supabase lets you write these in JavaScript without knowing SQL.

Content Script Chrome Extension

A JavaScript file in a Chrome Extension that runs inside the web page the user is viewing. It can read the page's content (DOM) and inject new UI elements into it.

manifest.json Chrome Extension

The control file every Chrome Extension must have. Tells Chrome the extension's name, what permissions it needs, which scripts run where, and what the popup UI file is called.

Service Worker Chrome Extension

The background script in a Chrome Extension (Manifest V3). Handles tasks like API calls, manages state, and coordinates messages between the popup and content scripts.

DOM General

Document Object Model. The browser's representation of a web page as a tree of elements. JavaScript can read and modify the DOM — this is how content scripts extract page data and how React updates the UI.

Workspace VS Code

The folder you've opened in VS Code. Everything inside it is your working project. VS Code treats the workspace as the root — search, file navigation, and Git all operate relative to it.

Extension (VS Code) VS Code

An add-on that gives VS Code new capabilities — Markdown preview, Git history, syntax highlighting for new file types, AI assistance. Installed from the Extensions panel with one click.
